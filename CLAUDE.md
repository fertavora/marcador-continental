## Repo layout (npm workspaces monorepo)

```
apps/web/       # Next.js 16 (App Router) web app
apps/mobile/    # Expo (SDK 57) + Expo Router mobile app, Android-first
packages/core/  # shared game logic/types (@marcador/core) — no DOM/RN deps, consumed as raw TS
```

`packages/core` holds only pure logic (`Game`/`Player` types, round rules, `createGame`/`addRound`/`endGame`/`getWinner`/`getTotals`, color assignment). Anything DOM-only (cookies) or RN-only (AsyncStorage) stays in each app's own `src/lib/storage.ts` — the two are separate implementations of the same shape, not shared code, because their persistence primitives are fundamentally different (synchronous cookies/localStorage vs async AsyncStorage backed by an in-memory cache).

## Architecture

### Web (`apps/web`)

- **Next.js 16 (App Router) + React 19 + TypeScript**, source under `apps/web/src/app`. Routes are file-based; this is a pinned major version with breaking changes from older Next.js — see the AGENTS.md import above before assuming API behavior from training data, and check `node_modules/next/dist/docs/01-app` for the current App Router docs when in doubt.
- **Styling: Tailwind CSS v4**, configured via `@import` directives directly in [apps/web/src/app/globals.css](apps/web/src/app/globals.css) (no `tailwind.config.*` file — v4 is CSS-first). Theme tokens (colors, radii) are defined there as CSS custom properties and consumed through the `@theme inline` block; dark mode is the `.dark` class variant.
- **UI components: shadcn/ui** (`style: base-nova`, base color `neutral`), installed via the `shadcn` CLI into `apps/web/src/components/ui`. It's built on `@base-ui/react` primitives (not Radix) and `lucide-react` icons. Add new primitives with `npx shadcn@latest add <component>` (run from `apps/web`) rather than hand-rolling them. Path aliases (`@/components`, `@/lib`, `@/hooks`, `@/components/ui`) are declared in [apps/web/components.json](apps/web/components.json) and map through the `@/*` TS path alias in [apps/web/tsconfig.json](apps/web/tsconfig.json).
- `apps/web/src/lib/utils.ts` exports the shadcn `cn()` helper (clsx + tailwind-merge) — use it for conditional class composition instead of manual string concatenation.
- `next.config.ts` sets `transpilePackages: ["@marcador/core"]` since that workspace package ships raw TS with no build step.

### Mobile (`apps/mobile`)

- **Expo SDK 57 + Expo Router + React Native 0.86 + TypeScript**, source under `apps/mobile/src/app`. This is a fast-moving, pinned SDK version — read `apps/mobile/AGENTS.md` (versioned Expo docs link) before assuming API behavior from training data.
- **Styling: NativeWind v4** (Tailwind classes on RN components), config in `apps/mobile/tailwind.config.js` + `apps/mobile/src/global.css` (processed via `apps/mobile/metro.config.js`'s `withNativeWind`). Tailwind v3 (not v4) on purpose — NativeWind's stable release targets v3's JS-config format, independent of the web app's Tailwind v4. Because of that version split, root `package.json` has an `overrides["@marcador/mobile"].tailwindcss` pin so npm nests mobile's v3 copy instead of letting hoisting collide with web's v4 (`@tailwindcss/postcss` resolves `tailwindcss` from its own directory upward, so a hoisting collision silently breaks whichever app didn't win the hoist).
- **No shadcn/base-ui here** — those are DOM-only. UI primitives are hand-rolled RN components in `apps/mobile/src/components` (`button.tsx`, `player-avatar.tsx`, etc.) styled with NativeWind classes. Dark mode has a manual toggle (`ThemeToggle` in the header, mirroring web) backed by NativeWind's `colorScheme` API and persisted via AsyncStorage; falls back to following the OS until the user overrides it — see `apps/mobile/src/lib/storage.ts`.
- Color tokens in `tailwind.config.js` are flat hex pairs (`background` / `background-dark`, etc.), not nested `{DEFAULT, dark}` objects — NativeWind's `dark:` variant only activates on separately-named utility classes, not nested color scale keys.
- Confirm dialogs use `Alert.alert` (native) instead of a custom `AlertDialog` component — simpler and feels native, at the cost of losing the custom destructive-red styling used on web.
- No cookie-consent modal — not applicable on native. Storage uses AsyncStorage directly (see below), gated only by the splash screen during initial hydration.

## State persistence

- **Web**: `apps/web/src/lib/storage.ts` — cookies for consent/theme, `localStorage` for games. All synchronous, read directly in `useSyncExternalStore` snapshots.
- **Mobile**: `apps/mobile/src/lib/storage.ts` — AsyncStorage is inherently async, so games live in an in-memory cache hydrated once at startup (`hydrate()`, called from `apps/mobile/src/app/_layout.tsx` before hiding the splash screen) and mutated synchronously thereafter, persisting to AsyncStorage in the background on every write. `useGames`/`useGame` return `undefined` until hydration completes.

## Product requirements (from REQS.md)

Marcador Continental is a scoreboard for the Spanish card game "Continental":

- Mobile-first, responsive, **UI text in Spanish**.
- Web: game state persists in browser cookies; the app must show a cookie-consent modal and cannot function if cookies are declined. Mobile: state persists via AsyncStorage, no consent modal (not applicable on native).
- A game has 7 rounds (each with a fixed combination name, e.g. "Dos tríos", "Tres escaleras" — see REQS.md) and 2–6 players; each player gets a name and a randomly assigned color at creation, and the game records its creation date/time.
- During a game, the user enters each player's points per round, including negative scores (e.g. -10).
- The app must show a history of past games.
